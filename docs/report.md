# Отчёт о разработке приложения lazy-sandbox методом контролируемого вайб-кодинга

---

## 1. Введение

Настоящий отчёт описывает процесс разработки веб-приложения **lazy-sandbox** — интерактивной песочницы для сравнения шаблонизаторов JavaScript. Приложение позволяет пользователю выбрать движок (Handlebars, Mustache, EJS), ввести шаблон и JSON-данные, получить результат рендеринга и сохранить замеренное время в базе данных через REST API.

Разработка велась в соответствии с принципом **контролируемого вайб-кодинга** — подходом, при котором разработчик формулирует задачу в виде чёткого текстового промпта, передаёт его ИИ-инструменту, критически оценивает результат, при необходимости корректирует и только после этого принимает сгенерированный код. Каждый промпт и его результат фиксируются в истории коммитов.

---

## 2. Принцип контролируемого вайб-кодинга

Термин «вайб-кодинг» описывает практику написания кода преимущественно через взаимодействие с языковыми моделями. «Контролируемый» вариант отличается от стихийного следующим:

- промпт формулируется до начала работы и содержит явные требования к архитектуре, поведению и качеству кода;
- сгенерированный результат проверяется разработчиком: запускается, тестируется, рецензируется;
- итоговый код коммитируется вместе с сохранённым промптом для воспроизводимости;
- инструмент выбирается осознанно под характер задачи.

Такой подход сохраняет скорость, характерную для ИИ-генерации, и при этом обеспечивает контроль над качеством и архитектурной целостностью проекта.

### 4.2 Этап 2. Разработка backend

**Дата:** 10 мая 2026  
**Автор:** is.bolshakov  
**Инструмент:** Claude  
**Коммит:** `feat(backend): benchmark storage API with SQLite`

**Промпт:**

> Реализуй FastAPI-сервис для хранения результатов бенчмарков шаблонизаторов.
>
> Требования:
> - Эндпоинты: `POST /benchmarks` (201, сохранить запись) и `GET /benchmarks` (200, список с фильтром по `engine`, пагинацией `limit`/`offset`).
> - Поля записи: `template_engine` (string, 1–64 символа), `render_time_ms` (float, ≥ 0), `payload` (text, опционально).
> - ORM: SQLAlchemy 2, синхронный режим, SQLite. Таблица создаётся при старте через `create_all`.
> - Валидация: Pydantic v2. Отрицательный `render_time_ms` — 422.
> - CORS настраивается через переменную окружения `CORS_ORIGINS`.
> - Конфигурация через `pydantic-settings` (файл `.env`).
> - Тесты: pytest + httpx TestClient, изолированная in-memory SQLite. Покрыть: health, валидацию, сортировку, фильтрацию, пагинацию.
> - Добавить `backend/README.md` и `.env.example`.

**Результат:**

Создано 18 файлов: `main.py` с lifespan и CORS middleware, `config.py`, `database.py` с поддержкой StaticPool для тестов, `models.py`, `schemas.py`, `crud.py`, роутер `benchmarks.py`. Написаны 9 тестов, покрывающих все ключевые сценарии. Swagger UI доступен на `/docs`. Сервис запускается одной командой `uvicorn app.main:app --reload`.

### 5.4 Наблюдения по методологии

**Что сработало хорошо:**

- Детальные промпты с явными архитектурными требованиями давали результат, близкий к production-quality с первой итерации.
- Сохранение промптов в сообщениях коммитов обеспечило полную воспроизводимость: любой участник команды может понять, почему код написан именно так.
- Разделение на этапы — каждый промпт решал одну чётко ограниченную задачу — позволяло легко проверять и принимать результаты.

**Что требовало доработки:**

- Базовый prompt для UI потребовал итерации: после первой генерации стандартный `<select>` имел проблемы с доступностью и стилизацией, что было исправлено отдельным промптом (этап 6).
- Статусы сохранения benchmark изначально были частично на английском языке — потребовалась корректировка промпта с явным требованием языка UI.

**Вывод:** контролируемый вайб-кодинг позволяет эффективно создавать качественный код при условии, что промпт содержит явные требования к архитектуре, типизации и поведению, а разработчик верифицирует каждый результат перед принятием. Инструмент при этом следует выбирать под характер задачи: Claude — для архитектурно сложных задач с большим контекстом, ChatGPT — для точечных шаблонных задач, Qwen — для UI-полировки.
